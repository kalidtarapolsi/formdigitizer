Key Insights

VBA forms have nearly 2x the fields — averaging 89 fields per form vs 52 on va.gov, making them higher-value targets for the composition engine.
XFA generic field names remain the #1 coverage blocker — field names like form1[0].#subform[0].TextField1[0] cannot be matched by name-based patterns. XFA label-proximity matching is the highest-impact improvement opportunity.
Signature is the most widespread pattern — matched across 273 forms (60% of all forms with fields), followed by Address (218 forms) and SSN (175 forms).
VBA forms show higher composition coverage — 36% vs 26% on va.gov, because VBA forms use more descriptive AcroForm field naming conventions.
31 forms achieve 76-100% coverage — these are ready for near-complete composition engine support today. Another 86 forms are at 51-75%, needing only modest pattern additions.

Recommended Next Steps

1. XFA Label-Proximity Matching: Build a secondary matcher that reads XFA XML label data (or uses spatial proximity to text labels in the PDF) to identify fields with generic names like TextField1[0]. This alone could recover 40-60% of currently unmatched fields.
2. Pattern Expansion: Add patterns for common unmatched field categories: military service dates, branch of service, discharge type, claim type, benefit election, relationship type, and education/training fields.
3. Spanish Form Variant Mapping: Many VBA forms have Spanish translations (e.g., 21P-0517-1 vs 21P-0517-1(Spanish)). Build a mapping that reuses the same composition template for both language variants.
4. Other-Domain Forms: Analyze the 21 forms hosted on opm.gov, gsa.gov, and other external domains via server-side proxy to complete full catalog coverage.
5. High-Coverage Form Templates: The 31 forms at 76-100% coverage are ready for composition engine template generation today. Start building templates for these to validate the end-to-end digitization pipeline.
