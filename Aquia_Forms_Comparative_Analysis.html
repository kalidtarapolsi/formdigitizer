Digitization Tiers

The analysis reveals three distinct tiers of VA form digitization:

TierDescriptionExampleWhat's Available
Full Digital Complete online form with validation, direct submission, and backend integration 21-0779, 21-0966 PDF + JSON schema + Online form + API integration
Upload + Schema Fillable PDF with digital upload path; schema exists for validation but no online form 21-2680 PDF + JSON schema + Upload endpoint
PDF Only Traditional fillable PDF, no digital infrastructure published 21-4192 PDF download only
Key Takeaway: Not all "digitized" forms follow the same path. Some forms (like 21-2680, which requires a physician's signature) may never have a full online form — instead they get a JSON schema for validation and an upload endpoint. Our digitizer should support generating outputs for all three tiers.

Common Patterns Across All Schemas

These patterns are consistent across 21-0966, 21-0779, and 21-2680 — they represent the VA.gov digital form conventions our digitizer should follow.

Pattern 1: Reusable Definitions

Both 21-0779 and 21-2680 use a definitions block with identical fullName and simpleAddress types. The simpler 21-0966 schema doesn't use definitions but follows the same field structure inline. Our digitizer should auto-detect common field groupings and generate reusable definitions.
DefinitionUsed InFieldsConstraints
fullName 21-0779, 21-2680, 21-0966 (inline) first (12), middle (1), last (18) middle is nullable; first/last required
simpleAddress 21-0779, 21-2680 street (30), street2 (5), city (18), state (2), postalCode (9), country (3) street/city/postalCode required; state required if USA/CAN/MEX

Pattern 2: Field Composition Rules

PDF forms use atomic fields (individual boxes). The VA.gov schema always composes them into these standard types:

PDF PatternSchema TypeFormat/PatternExample
SSN_1 (3) + SSN_2 (2) + SSN_3 (4) ssn → string ^\d{9}$ "123456789"
DOB_Month + DOB_Day + DOB_Year dateOfBirth → string format: "date" (ISO 8601) "1990-01-15"
First_Name + Middle_Name + Last_Name fullName → object $ref definition with maxLength { first, middle, last }
Street + Apt + City + State + ZIP + Country simpleAddress → object $ref definition with required fields { street, city, state, postalCode }
Phone_Area (3) + Phone_First (3) + Phone_Last (4) phoneNumber → string ^\d{10}$ "5551234567"
Multiple checkboxes (COMPENSATION, PENSION, etc.) benefitType/Selection → string enum: [...] "compensation" or "smc"
Multiple radio buttons (spouse, parent, child...) relationship → string enum: [...] "spouse"
Yes/No checkbox pairs → boolean type: "boolean" true / false

Pattern 3: Section-to-Property Mapping

PDF sections (I, II, III...) map to top-level schema properties. The naming convention varies by form but follows a consistent hierarchy:

PDF Section21-096621-077921-2680
Section I: Veteran ID veteran veteranInformation veteranInformation
Section II: Claimant/Dependent dependent claimantInformation claimantInformation
Section III: Form-Specific benefitType nursingHomeInformation benefitInformation
Section IV+: Additional — generalInformation additionalInformation
Signature (in veteran) (in generalInformation) veteranSignature
Naming Convention: The newer schemas (21-0779, 21-2680) use the xxxInformation suffix pattern consistently. The older 21-0966 schema uses shorter names. Our digitizer should default to the xxxInformation convention.

Pattern 4: Nullable vs Required

BehaviorSchema PatternExample
Required field "type": "string" + in required array ssn in 21-2680 veteranInformation
Optional field "type": ["string", "null"] middle name, vaFileNumber
Optional section oneOf: [{ type: "null" }, { type: "object" }] claimantInformation in 21-0779

Recommendations for Digitizer Tool

Based on analyzing how Aquia digitized these forms and the resulting VA.gov schemas, here are specific changes to make our form digitizer smarter.

1 Implement a Shared Definitions Library Critical

All VA.gov schemas reuse fullName and simpleAddress definitions. Our digitizer should auto-detect when extracted fields match these patterns and generate proper $ref definitions instead of inline objects. This is the single biggest structural difference between our current flat output and the real VA.gov schemas.

2 Build Field Composition Rules Engine Critical

The core value of the digitizer is mapping 61 atomic PDF fields → ~11 composite schema properties. Implement detection rules for: SSN parts → single string, date parts → ISO date, name parts → fullName object, address parts → simpleAddress object, checkbox groups → enum/boolean. These 6 rules cover ~80% of the composition needed.

3 Add Section → Property Mapping with xxxInformation Convention High

PDF sections (I, II, III...) should automatically map to top-level schema properties using the veteranInformation, claimantInformation naming convention. The digitizer already detects sections — it just needs to generate camelCase property names from them.

4 Support Three Output Tiers High

Not all forms become full online tools. The digitizer should generate: (1) PDF-faithful form (current behavior), (2) VA.gov-aligned JSON schema with definitions, and (3) Upload-ready HTML form. This matches the three tiers we observed (full digital, upload+schema, PDF only).

5 Add maxLength Constraints from PDF Field Properties Medium

PDF AcroForm fields have maxLength properties that the VA.gov schemas preserve (first name: 12, last name: 18, state: 2, etc.). Our digitizer already extracts these from pdf-lib — we just need to carry them through to the composed schema output.

6 Add Validation Pattern Generation Medium

The schemas include regex patterns for SSN (^\d{9}$), phone (^\d{10}$), and currency (^\d+(\.\d+)?$). Our digitizer should auto-generate appropriate validation patterns based on inferred field type.

7 Handle Nullable vs Required Correctly Medium

VA.gov uses ["string", "null"] for optional fields and oneOf: [null, object] for optional sections. Our digitizer should map PDF fields that are pre-filled or marked with asterisks as required, and others as nullable.

8 Side-by-Side Preview with Diff Highlighting Enhancement

Add a toggle between "PDF Faithful" and "VA.gov Schema" views in the preview tab, with visual indicators showing where fields were composed, renamed, or restructured. This helps form reviewers understand exactly what changed during the digitization process.
